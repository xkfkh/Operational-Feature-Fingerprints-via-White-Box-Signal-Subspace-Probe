# MSR / G1 白盒算法创新项目总规则

## 1. 项目总目标
当前项目的唯一主目标是：

围绕 `L1/run_g1_cr_gain.py` 这个 G1 / CRGain 基线，
在不引入黑盒神经网络头、不破坏白盒可解释性的前提下，
构造一个更强、更准、更稳定的白盒图学习算法。

## 2. 基线与数据
- 基线算法入口：`L1/run_g1_cr_gain.py`
- 数据目录候选（必须先自动确认真实使用路径，不允许主观假设）：
  - `planetoid/data`
  - `paper_data`
  - `论文数据`
  - `论文数据_v2`
- 新实验根目录：
  - `experiments_g1/wgsrc_development_workspace/`

## 3. 不可违反的硬约束
- 禁止把方案改成黑盒神经网络。
- 禁止引入 MLP、隐藏层黑盒分类头、端到端深层不可解释结构。
- 允许迭代、允许多阶段优化，但每一步更新必须有明确数学式。
- 所有核心模块必须是白盒、可解释、公式显式、变量含义明确。
- 可以使用：
  - 图拉普拉斯
  - 谱图理论
  - MCR² / coding rate / logdet
  - 低维线性子空间
  - 子空间角度 / 投影残差 / 类内离散度 / 类间分离度
  - 闭式或准闭式判别器
  - 白盒迭代算法
- 每次出现公式时，必须：
  1. 解释每个符号；
  2. 解释几何或信息论意义；
  3. 给一个小数值例子；
  4. 说明代码如何实现。

## 4. 多智能体协作的总原则
整个研发过程不是单智能体流程，而是：
- 每一个阶段都必须安排 **2 个智能体**
- 这 2 个智能体的 **目标任务固定，不允许被改写**
- 这 2 个智能体必须：
  - 各自独立思考
  - 提出算法
  - 提出理论理由
  - 做小规模实验
  - 对彼此结果进行质疑、比较和修正

同时，**不同阶段之间也必须允许交流**，因为：
- 某一阶段可用的算法，可能破坏其他阶段的需求；
- 某一阶段设计的输出形式，必须满足下一阶段输入要求；
- 某一阶段的量化指标，必须和最终预测器兼容。

因此，所有阶段必须遵守：

### 4.1 跨阶段信息同步
每个阶段开始前，都必须先读取：
- 前面阶段的结论
- 后面阶段的需求约束
- 当前全局接口说明

### 4.2 跨阶段交流机制
允许并要求：
- Phase 1 智能体与 Phase 2/3/4/5 的智能体交流
- Phase 2 智能体与 Phase 3/4/5 的智能体交流
- Phase 3 智能体与 Phase 4/5 的智能体交流
- Phase 4 智能体与 Phase 5 的智能体交流

交流目标不是改变对方任务，而是：
- 说明本阶段输出将如何影响下游
- 发现局部最优或模块冲突
- 修改接口、但不修改各自原始任务目标

### 4.3 固定任务，不允许漂移
每个智能体必须：
- 保持自己的原始职责不变
- 不允许中途转去做别的阶段的主任务
- 可以给别的阶段提建议，但不能替代别的阶段的主职责

## 5. 强制先做的事情
在任何创新前，必须先完成：
1. 读取并逆向分析 `L1/run_g1_cr_gain.py`
2. 找出：
   - 数据读取方式
   - 图预处理与拉普拉斯处理
   - 特征构造方式
   - 训练目标
   - 预测方式
   - 评估方式
3. 明确当前基线的：
   - 白盒程度
   - 数值瓶颈
   - 精度瓶颈
   - 可解释性瓶颈
4. 输出基线逆向分析笔记：
   - `experiments_g1/wgsrc_development_workspace/notes/00_baseline_reverse_engineering.md`
5. 做一次最小可运行复现，不先跳创新

## 6. 五个阶段的真实任务定义

---

### Phase 1：MCR²低维子空间构造
目标：
- 基于图拉普拉斯处理后的样本表示
- 把相似标签样本映射到可解释低维子空间
- 子空间构造要兼容后续阶段的差异量化、响应量化、预测器

要求：
- 不只是列方案，而是由 2 个智能体分别负责创新与实验
- 最终输出必须是：
  - 可实现的子空间构造算法
  - 清晰接口
  - 给后续阶段可直接调用的输出形式

---

### Phase 2：低维子空间内部元素差异量化
目标：
- 在 Phase 1 得到的低维子空间中，
  构造白盒算法来量化：
  - 样本之间差异
  - 各维度之间差异
  - 类内结构差异
  - 信息分布差异

要求：
- 这一步的算法输出必须能被 Phase 4 和 Phase 5 直接使用
- 做小规模实验验证差异量是否稳定、是否可解释

---

### Phase 3：量化不同子空间在实际 GNN 中的响应
这是纠正后的重点定义，不能写错。

目标：
- 不再是泛泛“影响度”
- 而是研究：
  **不同子空间在实际 GNN 传播、聚合、平滑、判别过程中，会产生什么响应**
- 构造白盒算法量化这种“子空间响应”

这里的“响应”包括但不限于：
- 某子空间在图传播后能量如何变化
- 某子空间在层间如何被放大、衰减、平滑
- 某子空间对类间可分性贡献如何变化
- 某子空间在实际 GNN 操作后残留了多少判别信息

要求：
- 必须给出显式响应量
- 必须可解释“为什么这个子空间在 GNN 中更敏感 / 更稳定 / 更有效”
- 必须与 Phase 1 子空间表示直接相连
- 必须与 Phase 5 最终预测器兼容

---

### Phase 4：量化子空间内部元素差距对结果的作用
目标：
- 不只是量化差异本身
- 而是量化：
  **子空间内部的元素差距，如何进一步作用到最终决策**
- 例如：
  - 类内离散度变化如何改变判别边界
  - 局部波动如何影响稳定性
  - 局部异常点如何改变结果
  - 子空间内部紧致度如何影响分类可信度

要求：
- 必须和 Phase 2 区分清楚：
  - Phase 2 是“差异是什么”
  - Phase 4 是“这些差异如何影响最终结果”
- 必须能输出给 Phase 5 使用的白盒量

---

### Phase 5：融合 Phase 3 和 Phase 4 构建最终白盒预测器
目标：
- 不直接用黑盒分类头
- 而是把：
  - Phase 3 的子空间响应量
  - Phase 4 的内部差距作用量
  结合成一个白盒预测器

要求：
- 预测器必须公式显式
- 必须能解释每项得分来源
- 必须说明为什么比 G1 / CRGain 更合理
- 必须说明为什么与前面阶段兼容

## 7. 多智能体研发结构
每个阶段都必须建立 2 个智能体，并遵守：

### 7.1 每个智能体必须完成的四件事
1. 理论分析
2. 白盒算法创新
3. 小规模实验
4. 对另一智能体和上下游阶段提出兼容性反馈

### 7.2 每个阶段必须输出四类内容
1. 数学定义
2. 算法实现建议
3. 小规模实验结果
4. 与前后阶段的接口兼容说明

### 7.3 跨阶段兼容性检查
每个阶段结束后，必须额外回答：
- 我的输出会不会破坏后续阶段？
- 后续阶段真正需要什么？
- 当前形式是否需要调整接口？
- 当前最优算法会不会牺牲整体最优？

## 9. 实验规则
- 先小规模实验，再扩大
- 小规模实验要求：
  - 小数据子集
  - 1~2 个随机种子
  - 少量迭代
  - 记录时间、稳定性、精度、解释性现象
- 所有实验都必须留日志
- 所有结论都必须回写到 notes

## 10. 输出语言与风格
- 默认中文
- 公式必须解释每个符号
- 不允许只给结论不解释
- 修改代码前必须先说明原因
- 必须重视整体逻辑链条，不允许只做局部 patch
